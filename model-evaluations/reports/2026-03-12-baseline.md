## Task 1 — Knowledge Graph Extraction

**Prompt:** Extract nodes+edges JSON from a paragraph about the Linux kernel, GNU project, and Systemd.
**Scoring (0-10):** 2pts nodes present, 2pts edges present, 2pts node schema valid, 2pts edge schema valid, up to 2pts for capturing key entities (Torvalds, Stallman, Poettering, Systemd, GNU).

All models produced schema-valid JSON. Qualitative differences:

| Model | Nodes | Edges | Notes |
|---|:---:|:---:|---|
| Claude | 14 | 12 | Best semantic typing. Temporal metadata in relation labels (`created_in_1991`). `kernel_space` modelled as Location. |
| `qwen3.5:35b-a3b` | 19 | 15 | Most comprehensive. Captured `kernel_space` as Location, all three components running within it. Full type taxonomy (`Function`, `Location`, `Architecture`). |
| `qwen2.5-coder:7b` | 14 | 11 | Surprisingly thorough for 7B. Included component-level nodes. Mixed-case IDs. Added unwanted markdown fences (stripped by eval). |
| `qwen3-coder:30b` | 13 | 12 | Modelled years (1991, 1983) as `date` nodes — valid design but unusual. Clean output, no fences. |
| `deepseek-r1:70b` | 12 | 10 | Also modelled years as nodes. Omitted component-level nodes. Clean valid JSON. |
| `qwen2.5-coder:32b` | 9 | 13 | Fewer nodes, more edges — collapsed components into relationships. Some edges referenced node IDs not in the nodes list. |
| `llama4:scout` | 9 | 8 | **Factual error:** `{"source": "Systemd", "target": "Linux kernel", "relation": "replaced"}` — Systemd replaced SysV init, not the kernel. Also duplicated the relationship bidirectionally. |

**Verdict:** For production KG extraction, use `qwen3.5:35b-a3b`. The 7B model is a good lightweight alternative. `llama4:scout` introduced a factual edge error.

## Task 3 — Reasoning / System Diagnosis

**Prompt:** A microservices platform shows latency spikes exclusively 02:00–04:00 UTC. Give 3 ranked root causes with specific diagnostic commands.
**Scoring (0-10):** Up to 3pts for 3 distinct numbered causes, 1pt each for time-related keywords (cron/batch/vacuum), connection pool keywords, diagnostic commands (backtick-quoted). Penalty for <80 words.

The correct answer: time-scheduled workloads (cron/batch/ETL), DB maintenance (autovacuum, checkpoints), Redis TTL thundering herd. The time-bounded clue is key.

**Claude — 10/10**
Identified all three root causes correctly. Provided `pg_stat_bgwriter` for checkpoint analysis and `redis-cli MONITOR` to sample cache misses in real time. Cause 3 (Redis thundering herd on TTL expiry) was the most insightful and was missed by all local models.

**`qwen3.5:35b-a3b` — 8/10**
Best local model. Identified DB maintenance, bandwidth saturation from data exports, and cloud storage I/O throttling as third cause. Provided `iftop`, `sar -n DEV`, `iostat -x` — all runnable and specific. Missing the Redis thundering-herd angle.

**`deepseek-r1:70b` — 8/10**
Correct top cause. Good structure. However, included a **hallucinated SQL query**: `SELECT * FROM pg_scheduled_tasks` — this table does not exist in standard PostgreSQL. Redis replication lag (cause 2) is less likely than thundering herd. The hallucination is subtle and production-dangerous.

**`qwen2.5-coder:7b` — 7/10**
Good top two causes. Cause 3 ("Network Latency Between Microservices") is the **symptom being described**, not a root cause. Suggested `kubectl` commands without Kubernetes being mentioned in the prompt.

**`llama4:scout` — 6/10**
First cause was JVM GC pauses — plausible in general but unjustified for this scenario (no JVM mentioned, and GC doesn't produce a strict 2-hour UTC window). Second cause was "Redis Cache Saturation" (a symptom). Third cause (PostgreSQL checkpoints) was valid.

**`qwen3-coder:30b` — 5/10**
Has good specific commands (`tcpdump`, `redis-cli info memory`) but cause 3 ("Network/Infrastructure Throttling") is too vague. Only 118 words — felt rushed. Fast MoE generation may have led to less deliberation.

**`qwen2.5-coder:32b` — 4/10**
Lowest score despite being a larger dense model than 7B. "Network Congestion During Maintenance" is accurate but not actionable. No actual PostgreSQL queries provided. `journalctl -u networkd-dispatcher` won't reveal DB-level root causes.

## Gaps vs Claude Baseline

1. **Structured output:** Local models are at parity. The gap is richness (fewer nodes/edges) not validity.
2. **Code generation:** Local models match Claude on correctness for well-defined algorithmic tasks. Gap widens on complex multi-file refactors or ambiguous specs.
3. **Reasoning:** Largest gap. Local models identify the right *category* of causes but miss precision. Key failure modes: hallucinated SQL tables (deepseek-r1), identifying symptoms as causes (llama4, 32b), assuming unstated infrastructure (7b with kubectl).
4. **Speed:** The MoE models (qwen3-coder, qwen3.5) are competitive with small dense models in tok/s while carrying much larger parameter counts — the right choice for this hardware.
